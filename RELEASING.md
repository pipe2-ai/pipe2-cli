# Releasing pipe2-cli

This directory ships as a separate public repo at
[`github.com/pipe2-ai/pipe2-cli`](https://github.com/pipe2-ai/pipe2-cli).
The files at the root of this directory — `go.mod`, `README.md`,
`AGENT.md`, `.claude-plugin/marketplace.json`, `.goreleaser.yml`, and
the `.claude/skills/pipe2-cli/` tree — are laid out so a plain
file-copy into an empty checkout produces a valid, releasable repo
with no further massaging.

## Automated release flow (default)

Pushing a `cli-vX.Y.Z` tag on the monorepo triggers
[`.github/workflows/publish-pipe2-cli.yml`](../../.github/workflows/publish-pipe2-cli.yml),
which mirrors this directory to `pipe2-ai/pipe2-cli`, tags it there,
and runs [goreleaser](../../packages/pipe2-cli/.goreleaser.yml) to
publish binaries + a Homebrew formula.

```bash
# From the monorepo root:
git tag cli-v0.1.0 -m "pipe2-cli v0.1.0 — initial release"
git push origin cli-v0.1.0
gh run watch --workflow=publish-pipe2-cli.yml
```

End-user install methods wired up by that one workflow:

```bash
brew install pipe2-ai/tap/pipe2                             # Homebrew
go install github.com/pipe2-ai/pipe2-cli/cmd/pipe2@latest   # Go toolchain
# or grab a tarball from github.com/pipe2-ai/pipe2-cli/releases
```

See [`docs/ops/pipe2-cli-release.md`](../../docs/ops/pipe2-cli-release.md)
for the one-time setup (repo creation, PAT, secret) and troubleshooting.

## Manual fallback (if CI is down)

```bash
# 1. Verify the CLI is clean locally.
cd packages/pipe2-cli
make docs-cli-codegen
go run ./cmd/gen-agent-md
go test ./...
cd ../..

# 2. File-copy mirror into a local clone of the public repo.
git clone git@github.com:pipe2-ai/pipe2-cli.git /tmp/pipe2-cli
cd /tmp/pipe2-cli
git rm -rf . 2>/dev/null || true
cp -r "$OLDPWD/packages/pipe2-cli/." .
git add -A && git commit -m "Release v0.1.0"
git tag -f v0.1.0 && git push origin main --tags --force

# 3. Run goreleaser from the mirror clone to publish binaries + formula.
GITHUB_TOKEN=$PAT HOMEBREW_TAP_TOKEN=$PAT goreleaser release --clean
```

## What happens to users

- **Claude Code, Codex CLI, ChatGPT Agents**: users run
  `/plugin marketplace add pipe2-ai/pipe2-cli` — Claude Code reads
  `.claude-plugin/marketplace.json` at the repo root, installs the
  `pipe2-cli` plugin, and the `SKILL.md` is active immediately.
- **OpenCLI-compatible runtimes**: agents discover the CLI via the
  `AGENT.md` at the repo root (also generated from the live cobra
  command tree, mirrored at every release).
- **Plain `go install`**: `go install github.com/pipe2-ai/pipe2-cli/cmd/pipe2@latest`
  still works because `go.mod` declares `module github.com/pipe2-ai/pipe2-cli`.

## Source of truth

Every file at the root of the public repo is generated from code in
this directory:

| File                                  | Source                                      | Regen command                |
|---------------------------------------|---------------------------------------------|------------------------------|
| `AGENT.md`                            | cobra command tree                          | `go run ./cmd/gen-agent-md`  |
| `.claude/skills/pipe2-cli/SKILL.md`   | `internal/cli/assets/SKILL.md` (embedded)   | `go run ./cmd/gen-agent-md`  |
| `.claude-plugin/marketplace.json`     | hand-written, version bumped per release    | edit by hand                 |
| `README.md`                           | hand-written                                | edit by hand                 |

CI enforces drift — see the `build-pipe2-cli` job in
`.github/workflows/deploy.yml` at the monorepo root.

## Why subtree split instead of a separate monorepo?

Keeping the CLI here during early development means:

- Tests run against the exact Hasura schema and api handlers in this
  same commit. No cross-repo integration gap.
- `packages/sdk-go` changes ship atomically with CLI changes that use
  them — no tedious version pinning.
- Docs generation, unit tests, and e2e tests all run in the same CI.

Once the surface stabilises and the release cadence decouples from
the monorepo, we can lift this directory into its own repo. The split
history is pre-flight: since every release is generated by
`git subtree split`, the public repo always has a clean, linear
history derived from these files.
