# Pipe2.ai cookbook

Canonical, runnable recipes for every tutorial published at
[pipe2.ai/recipes](https://pipe2.ai/recipes). Each recipe is a **typed
Go program** compiled into the `pipe2` binary, and is the single source
of truth for its tutorial: the article renders the recipe's generated
manifest, so editing the Go source updates both.

## Why this exists

We learned the hard way that prose-only tutorials drift away from the
live API. The `*_asset_url` → `*_asset_id` rename lived in the wild for
months in our published guides because nothing forced the prose and the
schema to agree. With recipes as real Go programs in the repo, a schema
change fails the recipe's test (or `make cookbook-check`) instead of
silently rotting a published guide.

## How a recipe is built

Each recipe lives at `cookbook/<slug>/` as its own Go package and
registers itself with the global registry via `init()` → `Register()`.
The CLI binary embeds every registered recipe at compile time, so
`pipe2 recipe run <slug>` is fully self-contained — no scripts, no
network fetch of the recipe itself.

The author writes two things in `recipe.go`:

- **`Manifest()`** — metadata + the chain shape (steps, inputs, sample
  provenance). Drives `pipe2 recipe info`, the article walkthrough, and
  the generated `recipe.json`.
- **`Run(ctx)`** — the actual orchestration: typed inputs, pipeline
  invocations, output capture, error handling.

## Layout

Each recipe directory contains:

| File | Audience | Source of truth? |
|---|---|---|
| `recipe.go` | The runnable — `Manifest()` + `Run()` | ✅ hand-written |
| `recipe.json` | Website + `pipe2 recipe info` manifest | ❌ **generated** from `Manifest()` — never hand-edit |
| `README.md` | The narrative the data can't carry (the *why*) | ✅ hand-written, see [`AUTHORING.md`](./AUTHORING.md) |
| `recipe_test.go` | Guards the manifest/chain against schema drift | ✅ hand-written |
| `samples/` | Frozen per-step outputs the article previews | generated by a captured run |

`recipe.json` is regenerated from the Go source — see the
[Cookbook recipe.json](../../../CLAUDE.md) rule. Run `make cookbook-regen`
to rebuild it; `make cookbook-check` is the CI/pre-commit drift gate.

## How recipes are consumed

- **Humans & AI agents**: `pipe2 recipe list`, `pipe2 recipe info <slug>`,
  then `pipe2 recipe run <slug> --<input> <value> …`. The binary ships
  every recipe, so nothing is cloned or pasted.
- **Published tutorials at [pipe2.ai/recipes](https://pipe2.ai/recipes)**:
  render the generated `recipe.json` (chain walkthrough, inputs, cost,
  sample previews) alongside the hand-written `README.md` narrative — so
  the article and the runnable can never drift.

## Adding a new recipe

1. Create `cookbook/<slug>/` as a Go package implementing `Manifest()`
   and `Run(ctx)`, and `Register()` it from `init()`.
2. Add the package to the blank-import list in
   [`all/all.go`](./all/all.go) so the binary compiles it in.
3. Run the recipe end-to-end against the live API at least once and
   freeze its step outputs into `samples/`.
4. Run `make cookbook-regen` to generate `recipe.json`.
5. Write `README.md` following [`AUTHORING.md`](./AUTHORING.md).
6. Update the index above.

Recipes are self-contained: they never reference paths outside this
directory. That keeps the cookbook portable and publishable on its own
once the cross-repo migration lands.
